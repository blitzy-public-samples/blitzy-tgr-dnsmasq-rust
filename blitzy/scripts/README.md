# Config I SonarQube scan scripts

This directory holds the auditable implementation of the user-mandated
five-directive (D1..D5) SonarQube Community Build workflow for the
`blitzy-tgr-dnsmasq-rust` repository. The scripts produce
`findings-config-i.json` at the repository root.

The presence of these files is the response to the Checkpoint 1 MAJOR
review finding, which asked for the normalization and host-workflow
implementation to be made available for inspection so that the
Compliance Matrix safeguards (rows 16..26) can be verified by reading
code rather than only by inspecting the resulting JSON artifact.
Rows 25 and 26 were added in response to the subsequent QA pass that
identified upstream rolling-tag drift (the scanner now requires token
auth) and the absence of a `cargo` PATH guard.

## Files

| File | Role |
|------|------|
| `run_sonarqube_scan.sh` | Bash driver. Implements D1 (host tooling install + image pull), D2 (container start + cold-start poll), D3 (sonar-scanner invocation with token-based auth — see "Authentication" below), and the explicit D5 teardown. Registers the idempotent teardown trap at the top of the script. |
| `sonar_pipeline.py` | Python pipeline. Implements D4 (paginated `/api/issues/search` harvest with the `paging.total >= 10000` fallback) and D5 (severity remap, CWE cascade with memoization, `.rs` path filter, positive-integer line filter, code-point truncation, minified JSON serialization). Exposes a `--dry-run-issues` mode for offline branch coverage. |
| `sonar-project.properties` *(gitignored)* | Transient mirror of the user-supplied properties. Regenerated by `run_sonarqube_scan.sh` at execution time for human audit visibility only — the scanner is invoked with every property passed via `-D` flags, so the file is never read at runtime. The file is gitignored per AAP §0.6.2.4 so each run leaves the working tree byte-identical to its pre-run state. |

The application source tree (`src/`, `tests/`, `benches/`, `examples/`,
`Cargo.toml`, `Cargo.lock`, `rust-toolchain.toml`, `rustfmt.toml`,
`clippy.toml`, `.cargo/config.toml`, `build.rs`, `README.md`, `docs/`,
`blitzy/documentation/`) is read-only for the workflow and is not touched
by any script in this directory.

## Running

The host must have Docker (`>= 20.10`), `sonar-scanner` (apt-installable),
`python3` (`>= 3.10`), and `curl` available. From the repository root:

```
bash blitzy/scripts/run_sonarqube_scan.sh
```

The driver:

1. Sources `$HOME/.cargo/env` so that `cargo` is reachable by the SonarQube
   Rust analyzer (AAP §0.8.1 requirement; the rustup env file is idempotent).
2. Installs `sonar-scanner` if it is not already on `PATH`.
3. Pulls `sonarqube:community` and records the image-pull duration.
4. Launches the container detached as `sonarqube-test` on host port `9000`.
5. Polls `http://localhost:9000/api/system/status` once per second until
   the server reports `UP`, with a hard 120 s timeout. Accepted transient
   states are `STARTING`, `DB_MIGRATION_NEEDED`, and `DB_MIGRATION_RUNNING`.
   `DOWN` fails fast.
6. Mints a short-lived `GLOBAL_ANALYSIS_TOKEN` from the user-supplied
   `admin/admin` credentials via `POST /api/user_tokens/generate`
   (see "Authentication" below).
7. Runs `sonar-scanner` with the project key, the absolute source path,
   the host URL, the freshly minted token (as `sonar.token`), and
   `sonar.qualitygate.wait=true`.
8. Invokes `sonar_pipeline.py` to harvest issues (using `admin/admin`
   Basic auth, which the SonarQube REST API still accepts) and write
   `findings-config-i.json`.
9. Stops and removes the container via the top-of-script exit trap. The
   token is destroyed alongside the container's H2 database.

## Authentication

The user-supplied `admin/admin` credentials are the authoritative
authentication input. Current `sonarqube:community` rolling-tag releases
(>= v26.5.0) reject `sonar.login`/`sonar.password` in the *scanner*
protocol with `"Not authorized. Please check the user token in the
property 'sonar.token' or 'sonar.login' (deprecated)."` while continuing
to accept `admin/admin` Basic auth on the REST API. The driver therefore:

* Uses `admin/admin` Basic auth on `POST /api/user_tokens/generate` to
  mint an ephemeral `GLOBAL_ANALYSIS_TOKEN` named
  `config-i-scan-<unix-epoch>`.
* Passes that token to `sonar-scanner` as `-Dsonar.token=<TOKEN>`.
* Uses `admin/admin` Basic auth in `sonar_pipeline.py` for the
  `/api/issues/search` (and, when needed, `/api/rules/show`) calls — the
  REST API has not deprecated Basic auth.
* Destroys the token along with the container at teardown. The token's
  lifetime is therefore strictly bounded by the run duration (typically
  under five minutes), preserving the AAP §0.8.1 ephemeral-credentials
  risk-mitigation chain: "the entire SonarQube state is destroyed by the
  teardown step. This is the safest behavior for an ephemeral run."

The token VALUE is never echoed to the console or written to disk; only
the token NAME (a non-secret identifier) is recorded in the run
telemetry summary at end of run so the decision log can correlate a run
to its scanner-authentication artifact.

## Dry-run validation

`sonar_pipeline.py` accepts `--dry-run-issues <path>` and
`--dry-run-rules <path>` to substitute a pre-recorded issue payload and a
fixture map for the CWE cascade fallback. This allows every normalization
branch to be exercised without a live SonarQube instance:

```
python3 blitzy/scripts/sonar_pipeline.py \
  --dry-run-issues fixtures/raw-issues.json \
  --dry-run-rules  fixtures/rule-shows.json \
  --project-key    blitzy-tgr-dnsmasq-rust \
  --output         /tmp/findings.json
```

## Audit cross-reference — Checkpoint 1 AAP Compliance Matrix

The original MAJOR finding on `findings-config-i.json` enumerated nine
rows (16..24) that the reviewer could not verify from the JSON artifact
alone, and the follow-up QA pass added rows 25 and 26 for the cargo PATH
guard and the scanner-protocol token-auth migration. Each row maps to a
specific tagged code location below.

| Row | Requirement | Code location |
|-----|-------------|---------------|
| 16 | Severity remap dictionary implemented exactly, unknown severities hard-error | `sonar_pipeline.py` — `SEVERITY_REMAP` (module constant) and `normalize_severity()` raise `ValueError` on unknown input |
| 17 | CWE cascade implemented in order with memoized rule lookup | `sonar_pipeline.py` — `CweResolver.resolve()` (tag scan via `CWE_TAG_RE`, fallback via `_lookup_rule_cwe()` which caches in `self._cache`, sentinel `CWE_UNKNOWN`) |
| 18 | Component prefix stripping and `.rs` suffix filtering implemented | `sonar_pipeline.py` — `normalize_component()` strips the `<project_key>:` prefix and rejects non-`.rs` paths |
| 19 | Positive integer line filtering implemented | `sonar_pipeline.py` — `normalize_line()` returns `None` for missing, non-integer, zero, and negative line values |
| 20 | Description truncation uses Unicode code-point slicing and no ellipsis | `sonar_pipeline.py` — `truncate_description()` slices on the `str` (code points), with no ellipsis appended |
| 21 | SonarQube issue API uses exact type filter and paginates with `ps=500` | `sonar_pipeline.py` — `ISSUE_TYPES` (`"VULNERABILITY,BUG"` literal), `ISSUES_PAGE_SIZE_DEFAULT = 500`, and `fetch_issues_page()` constructs the query verbatim |
| 22 | `paging.total >= 10000` fallback exists | `sonar_pipeline.py` — `harvest_issues()` compares `total` against `ISSUES_PAGE_SIZE_HARD_CAP` and falls through to per-subcomponent harvesting via `_harvest_for_component()` |
| 23 | Cold-start polling uses required status handling and timeout | `run_sonarqube_scan.sh` — `COLD_START_TIMEOUT_S=120`, status `case` block handles `UP`/`DOWN`/`STARTING`/`DB_MIGRATION_*`/unknown explicitly, curl carries `--connect-timeout` and `--max-time` |
| 24 | Idempotent teardown trap registered at top of implementation | `run_sonarqube_scan.sh` — `trap '...docker stop...docker rm -f...' EXIT INT TERM HUP` immediately after the script header |
| 25 | Cargo on PATH for full Rust analysis (AAP §0.8.1) | `run_sonarqube_scan.sh` — sources `$HOME/.cargo/env` immediately after `set -o pipefail`; rustup's env file is idempotent so the source is a no-op when cargo is already on the system PATH |
| 26 | Scanner authentication compatible with current SonarQube Community Build (AAP §0.8.1 rolling-tag drift) | `run_sonarqube_scan.sh` — mints an ephemeral `GLOBAL_ANALYSIS_TOKEN` via `POST /api/user_tokens/generate` (`admin/admin` Basic auth) immediately after the cold-start poll; passes the token via `-Dsonar.token`; token is destroyed at container teardown |

## Reproducibility

Running the driver against an unchanged repository, network, and Docker
Hub state will reproduce the byte-identical `findings-config-i.json`
committed in the parent directory of this scripts folder. The run
captured in commit `140e2e9` produced `[]\n` (3 bytes) because the
SonarQube Community Build's Rust analyzer found zero `VULNERABILITY` or
`BUG` issues in the workspace, consistent with the project's own
self-reported 0/0/0 issue counts at the current 90% milestone.
