Section 03

Why these decisions

The decision log decisions-config-h.md carries the full table. Three deviations are worth surfacing for executive review.

Why the SBOM fallback

snyk test --all-projects does not parse Cargo manifests; it returns Snyk error SNYK-CLI-0008.

  • Generated CycloneDX 1.5 JSON with cargo cyclonedx: 240 components, all identified by pkg:cargo/... purls.
  • Consumed via snyk sbom test, which resolves cargo purls against Snyk's Rust advisory database.
  • Output schema for results-snyk-deps.json is identical to snyk test --json: a top-level vulnerabilities array.
  • Severity threshold flag is honoured by snyk sbom test so the intermediate artifact is already filtered.

SAST coverage interpretation

Snyk Code returned an empty SARIF: 0 results, 0 rules. The SARIF coverage block lists only .html as a loaded language pack.

Most likely cause: Snyk Code's Rust support is in the Early Access tier and is not enabled for this organisation. The pipeline accepts the empty SARIF as authoritative for the SAST contribution and surfaces no false positives.

Decision log entry: "Empty SAST result interpretation".

Section 04

Risks and next steps

Where the deliverable is silent or partial, where remediation belongs, and what a follow-up Config run would change.

Residual risks

  • Snyk's Rust SAST rule pack is gated; an Early Access entitlement would surface findings this run cannot.
  • Severity threshold high excludes low and medium advisories from the deliverable.
  • SBOM generation reflects Cargo.lock at scan time; Snyk's advisory database is mutable, so re-runs may add findings without code changes.
  • The Snyk "Forbidden" telemetry post on stderr is non-fatal; it does not affect scan output but is preserved in audit logs.
