## Deviations from Literal Requirements

The workflow diverged from the literal user prompt in the following places. Each entry uses the format `Deviation · Justification`.

- **Three sibling artifacts rather than one** · The user statement `~0 files modified | 1 new file` names `findings-config-i.json` only. The Explainability rule (AAP §0.7.1) and the Executive Presentation rule (AAP §0.7.2) apply globally to every deliverable on the Blitzy platform and add two further required files (`decision-log-config-i.md`, `executive-summary-config-i.html`), per the reconciliation note in AAP §0.7.4.
- **Scanner authentication via a short-lived token instead of `sonar.login=admin` / `sonar.password=admin`** · SonarQube Community Build `26.5.0.122743` deprecated username-password authentication in the scanner protocol (captured verbatim in `blitzy/qa-evidence/checkpoint1-evidence.txt`: `Not authorized. Please check the user token in property 'sonar.token' or 'sonar.login' (deprecated).`). The workflow uses the user-supplied `admin/admin` credentials as the only authentication input, mints an ephemeral token from those credentials via `POST /api/user_tokens/generate`, and destroys both the token and the container at teardown.
- **No rule-description fallback was exercised and no `CWE-UNKNOWN` sentinel was emitted** · The raw issue count was `0`, so neither the tag scan nor the `/api/rules/show` fallback ran. The count of `CWE-UNKNOWN` records is therefore `0`. The cascade code path is exercised end-to-end by adversarial fixtures in `blitzy/qa-evidence/checkpoint1-evidence.txt` (3 rule lookups across 4 issues, 2 memoized hits), confirming the cascade behaves correctly on synthetic data with all three branches reachable.
- **`cargo` exposed on `PATH` before scanner invocation** · Directive D3 does not mention `cargo` PATH management, but the Rust analyzer full ruleset requires `cargo` on `PATH` per AAP §0.8.1. Sourcing `$HOME/.cargo/env` at the top of `run_sonarqube_scan.sh` leaves the literal scanner invocation byte-identical while ensuring the Clippy sensor (the bulk of the 85-rule Rust ruleset) actually runs.
- **Mermaid version pin moved from 11.4.0 (AAP §0.7.2 literal) to 11.15.0** · The Checkpoint 2 review classified `mermaid@11.4.0` as a known-vulnerable dependency (CVE-2026-41150 and several injection/XSS classes fixed at 11.15.0). The upgrade is a security mitigation that overrides the literal version pin in AAP §0.7.2; the pinned SRI hash now captures the exact 11.15.0 bytes so the deck remains reproducible.
- **`securityLevel` lowered to `strict` (was `loose`)** · The previous configuration set `securityLevel: 'loose'`, which allows arbitrary HTML and JavaScript in Mermaid node labels. The Checkpoint 2 review flagged this as a security hardening gap. `strict` is now the active value; node labels in this deck contain only plain text, so the change has no visual effect but closes the injection surface.
- **Subresource Integrity (SRI) added to every external CDN tag** · The AAP did not mandate SRI but the Checkpoint 2 review asked the deck to consider it. SRI sha384 hashes were derived locally from the pinned CDN URLs (reveal.css, theme/white.css, reveal.js, mermaid.min.js, lucide.min.js) and attached to each tag along with `crossorigin="anonymous"` and `referrerpolicy="no-referrer"`. The deck remains a single-file deliverable and now refuses to evaluate tampered CDN bytes.
- **Decorative Lucide icons annotated `aria-hidden="true" focusable="false"`** · The AAP did not mandate the attributes but the Checkpoint 2 review identified the gap. Every `<i data-lucide="…">` placeholder in the deck now carries both attributes, so screen readers skip the decorative icons and keyboard tab order is unaffected.
- **Slide-body text trimmed to ≤ 40 words on every content slide** · The Executive Presentation rule in AAP §0.7.2 caps body text at 40 words. Several content slides previously exceeded this; trimming was applied to slides 3, 5, 8, 9, 11, 12, and 14. Counts verified by a static word-count tool: every content slide is now ≤ 40 words.
- **Workflow infrastructure files retained under `blitzy/scripts/` and `blitzy/qa-evidence/`** · The Checkpoint 2 review identified these files as out-of-scope for the three-sibling-artifact bundle. They are nevertheless retained as the auditable record of how `findings-config-i.json` was produced (driver + Python pipeline implementing D1 through D5) and the empirical basis of every numeric value in this decision log and the executive summary. The deviation is explicit; a future maintainer who wants only the three deliverables can delete the directories without affecting the three sibling artifacts.
- **`findings-config-i.json` empty form is `[]\n` (3 bytes)** · AAP §0.5.6.5 says the empty sentinel "writes the literal two-byte sequence `[]`", but the controlling pass criterion in AAP §0.1.3 is `wc -l = 1` and POSIX `wc -l` returns `0` for a 2-byte file without a trailing newline. The contradiction is resolved by following the pass criterion: the file ends with `\n` so `wc -l` returns `1`. AAP §0.5.6.5 itself acknowledges this reading ("newline terminator at end of file is permitted by `wc -l` returning `1`"). The 3-byte form is reproduced and verified in `blitzy/qa-evidence/checkpoint1-evidence.txt` §2.4.

## Baseline Contextualization

The empty findings set is consistent with the project pre-existing security posture. The Rust reimplementation enforces memory safety through the language ownership model, parses protocol input with `nom 7.1` ([Cargo.toml:L52]), and performs DNSSEC cryptography with `ring 0.17` ([Cargo.toml:L45]). The repository ships zero compilation errors and zero strict clippy warnings against the relaxed thresholds in `clippy.toml` (cognitive-complexity 30 at [clippy.toml:L9], too-many-arguments 8 at [clippy.toml:L17], too-many-lines 150 at [clippy.toml:L21]), and 592 of 592 tests pass per AAP §0.2.3. The SonarQube Rust analyzer (Clippy-based, 85 rules per AAP §0.2.2) runs on top of that baseline; a `[]` result reflects that the Clippy-derived rule set finds nothing of severity `VULNERABILITY` or `BUG` that the project own strict clippy gate has not already addressed. The scan window also covered the `tests/`, `benches/`, and `examples/` trees (10 additional Rust files), and no finding surfaced in those directories either.
